# 027 — Optional Initialization Verification

## Feature Name
Optional Initialization Verification

## Feature ID
027

## Status
Proposed

## Summary
Establish the official smoke-verification path for initialization in `amazing-async-dev`, covering both:

1. **Direct / manual initialization**
2. **Starter-pack initialization** using a compatible `starter-pack.yaml` generated by ecosystem tools such as `amazing-skill-pack-advisor`

This feature does **not** introduce a new execution capability. It consolidates and productizes the existing cross-repo integration work into a low-friction verification entry so users can quickly determine whether the initialization flow is actually working end to end.

This feature follows the positioning established in **026 — Optional Advisor Integration Positioning & Initialization Modes**:

- `amazing-async-dev` must remain fully usable without advisor.
- `amazing-skill-pack-advisor` is a **first-party optional ecosystem provider**, not a required dependency.
- Integration is defined at the **starter-pack contract** layer, not at a repo-coupled implementation layer.

---

## Problem
The ecosystem now has the core pieces needed for optional starter-pack initialization:

- `amazing-skill-pack-advisor` can produce starter packs with integration metadata and async-dev compatibility fields.
- `amazing-async-dev` can consume starter packs via `new-product create --starter-pack ...`.
- Pilot validation, onboarding improvements, examples, and docs already exist.

However, the current experience is still fragmented:

- Users must piece together validation steps across README, quick-start docs, pilot docs, examples, and tests.
- It is still harder than it should be to answer: **“Did this cross-repo initialization path actually work?”**
- The optional nature of advisor integration needs a matching verification experience that clearly validates both supported modes:
  - direct/manual mode
  - starter-pack mode

The result is that the ecosystem is technically more complete than it feels operationally.

---

## Goal
Create a clear, official, low-cognitive-load verification entry for initialization so a user can quickly confirm:

- direct/manual initialization works
- starter-pack initialization works
- compatible starter packs are accepted and mapped correctly
- incompatible or malformed starter packs fail clearly
- the optional advisor path is validated without becoming mandatory

---

## Non-Goals
This feature does **not**:

- make advisor a required dependency for async-dev
- add a new planner / recommendation capability
- introduce a large workspace UI or dashboard
- redesign the starter-pack contract
- replace existing onboarding docs wholesale
- expand execution-policy or async-decision capabilities

---

## Product Positioning

### Core Principle
`amazing-async-dev` must remain fully usable without advisor.

### Supported Initialization Modes
1. **Direct mode**
   - User initializes directly inside async-dev.
   - No ecosystem provider required.

2. **Starter-pack mode**
   - User initializes from a compatible `starter-pack.yaml`.
   - `amazing-skill-pack-advisor` is the first-party recommended ecosystem provider for generating such files.
   - Other future providers may also be supported if they satisfy the contract.

### Verification Principle
The official verification path must validate both supported modes without implying that starter-pack mode is the only “real” path.

---

## User Stories

### US-1 — Direct Mode Confidence
As a new async-dev user, I want a short official verification path for direct/manual initialization so I can confirm the core product works without depending on other repos.

### US-2 — Starter-Pack Confidence
As an ecosystem user, I want a short official verification path for starter-pack initialization so I can confirm a generated starter pack is actually consumable by async-dev.

### US-3 — Fast Failure Diagnosis
As an operator, I want verification output and documentation to tell me why initialization failed so I can quickly fix version mismatch, malformed files, or missing fields.

### US-4 — Correct Ecosystem Positioning
As a reader of the docs, I want the verification path to make it obvious that advisor is optional, so I understand the architecture and do not confuse the ecosystem plugin with a required dependency.

---

## Scope

### In Scope
#### `amazing-async-dev`
- Add a single official verification entry for initialization.
- Provide a smoke-verification path for:
  - direct/manual initialization
  - starter-pack initialization
- Document expected success markers and common failure modes.
- Link verification clearly from README and onboarding entry points.
- Optionally add a lightweight CLI verification surface if it stays narrow and easy to maintain.

#### `amazing-skill-pack-advisor`
- Add a small documentation bridge pointing users to the async-dev verification flow.
- Clarify that advisor is an optional first-party provider for starter-pack mode.

### Out of Scope
- New multi-provider plugin system implementation
- Rich status dashboard / workspace UI
- Full repo orchestration layer
- New starter-pack authoring workflows

---

## Proposed Deliverables

### Primary Deliverables in `amazing-async-dev`
1. `docs/verify.md`
   - the official verification entry
2. `examples/verify-initialization/`
   - minimal verification-oriented example assets and walkthrough
3. README updates
   - clear links to verification entry
   - clear distinction between direct mode and starter-pack mode

### Optional Deliverable
4. Lightweight verification command, only if implementation remains small and stable, for example:

```bash
asyncdev verify initialization --starter-pack path/to/starter-pack.yaml
```

This command is optional for 027. The feature is still complete if the official verification path is delivered through docs + example + linked workflow.

### Supporting Deliverables in `amazing-skill-pack-advisor`
1. README / docs usage note:
   - how to verify exported starter packs with async-dev
2. Cross-link to async-dev verification entry

## Functional Requirements

### FR-1 — Official Verification Entry Exists
The repo must provide one clearly discoverable official verification entry for initialization, rather than requiring users to reconstruct validation from scattered docs.

### FR-2 — Direct Mode Verification Exists
The verification entry must include a minimal direct/manual initialization smoke path.

### FR-3 — Starter-Pack Mode Verification Exists
The verification entry must include a minimal starter-pack initialization smoke path using a compatible `starter-pack.yaml`.

### FR-4 — Optional Positioning Is Explicit
The verification materials must explicitly state:
- direct mode is fully supported
- starter-pack mode is optional
- advisor is a first-party ecosystem provider, not a required dependency

### FR-5 — Success Markers Are Concrete
The verification flow must tell the user what “working” looks like, including expected artifacts, states, or outputs.

### FR-6 — Failure Modes Are Actionable
The verification flow must describe common failures and next actions, including at minimum:
- incompatible starter-pack version
- malformed starter-pack file
- missing required fields
- initialization failure due to path or command usage issues

### FR-7 — Cross-Repo Discoverability Exists
Advisor docs must point users to the async-dev verification path for starter-pack consumption.

---

## UX Requirements

### UX-1 — Low Cognitive Load
A user should be able to understand the difference between:
- direct mode
- starter-pack mode
- optional ecosystem integration

without reading deep architecture docs.

### UX-2 — Fast Path First
The verification path should be optimized for fast confirmation, not comprehensive theory.

### UX-3 — Clear Outcomes
Each verification path should end in a clear result shape such as:
- pass / success markers seen
- fail / likely cause
- next action

### AC-3 — Starter-Pack Mode Coverage

## Risks

### Risk 1 — Overbuilding Into a New CLI Surface
The feature could become unnecessarily large if it tries to solve too much through commands instead of a focused verification entry.

**Mitigation:**
Keep 027 documentation-first. Treat CLI verification as optional and narrow.

### Risk 2 — Reintroducing Advisor Coupling
The feature could accidentally imply that starter-pack mode is the preferred or only serious path.

**Mitigation:**
Always present direct mode and starter-pack mode as parallel supported modes.

### Risk 3 — Duplicating Existing Docs Without Consolidation
The feature could add more docs without actually reducing confusion.

**Mitigation:**
Require a single official verification entry and cross-link existing material into it rather than scattering new content.
