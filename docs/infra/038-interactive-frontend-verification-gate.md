# Feature 038: Interactive Frontend Verification Gate

> Prevent false "tested" outcomes when only frontend server has been started.

---

## Metadata

| Field | Value |
|-------|-------|
| feature_id | `038-interactive-frontend-verification-gate` |
| title | Interactive Frontend Verification Gate |
| goal | Ensure browser-level verification is mandatory for interactive frontend work, preventing false completion when only server startup occurred |
| user_value | Operators can trust that frontend features with interaction were actually verified at browser level, not just "server started" |
| status | planning |

---

## Problem Statement

**Current undesired pattern:**
1. Interactive frontend feature implemented
2. App server started
3. Test phase reported or implied
4. Browser automation not actually executed
5. Task appears more validated than it really is

**Core principle:**
> For interactive frontend work, environment setup is not verification.
> Server startup is a prerequisite step, not a validation result.

---

## Goal

Introduce an explicit **Interactive Frontend Verification Gate** so that:

- Frontend validation does not terminate at server startup
- Browser-level verification becomes mandatory for interaction-oriented work unless explicitly impossible
- Playwright or equivalent browser automation is invoked when required
- Setup, readiness, execution, and evidence capture are treated as distinct stages
- Test completion cannot be claimed without verification evidence or an explicit exception reason

## Schema Field Definitions

### verification_type (ExecutionPack)

```yaml
verification_type:
  type: string
  description: Classification of verification requirements
  enum:
    - backend_only           # No frontend, no browser verification needed
    - frontend_noninteractive # Frontend but no user interaction (e.g., static display)
    - frontend_interactive   # User interaction required (clicks, forms, navigation)
    - frontend_visual_behavior # Visual state transitions, animations
    - mixed_app_workflow     # Backend + frontend interaction flow
  default: backend_only
  required: false
```

### browser_verification (ExecutionResult)

```yaml
browser_verification:
  type: object
  description: Browser-level verification evidence
  required: false
  fields:
    executed:
      type: boolean
      description: Whether browser verification was actually executed
    passed:
      type: integer
      description: Number of scenarios that passed
    failed:
      type: integer
      description: Number of scenarios that failed
    skipped:
      type: integer
      description: Number of scenarios skipped
    exception_reason:
      type: string
      description: Structured reason if browser verification could not run
      enum:
        - playwright_unavailable
        - environment_blocked
        - browser_install_failed
        - ci_container_limitation
        - missing_credentials
        - deterministic_blocker
        - reclassified_noninteractive
    scenarios_run:
      type: array
      description: List of scenarios that were executed
      items_type: string
    screenshots:
      type: array
      description: Paths to screenshot artifacts
      items_type: string
    duration:
      type: string
      description: Time spent on browser verification
```

## AGENTS.md Additions (Section 9)

```markdown
## 9. Interactive Frontend Verification Gate

### 9.1 Classification Rule
When executing a feature/task, determine `verification_type`:
- `backend_only`: No browser verification required
- `frontend_noninteractive`: Browser verification optional
- `frontend_interactive`: Browser verification mandatory
- `frontend_visual_behavior`: Browser verification mandatory
- `mixed_app_workflow`: Browser verification mandatory for frontend portion

### 9.2 Completion Gate (FR-7)
A frontend-interactive task MUST NOT be marked `status: success` unless:
1. `browser_verification.executed: true` and results recorded
2. OR valid `browser_verification.exception_reason` recorded

### 9.3 Playwright Invocation Policy (FR-9)
When `verification_type` requires browser verification:
- MUST invoke Playwright skill (`/playwright`) after server startup
- MUST NOT stop at "server ready" without browser run
- MUST capture evidence (scenarios, screenshots)

### 9.4 Orchestration Continuation (FR-10)
After server startup for frontend verification:
- MUST continue to browser verification stage
- MUST handle long-running server processes
- MUST poll for server readiness before Playwright launch
```

## External Tool Engine Markdown Template Addition

When `verification_type` is `frontend_interactive`, `frontend_visual_behavior`, or `mixed_app_workflow`:

```markdown
## Browser Verification (MANDATORY)

**This task requires browser-level verification.**

### Stages
1. Build/setup â†’ Complete build steps
2. Server start â†’ Start local dev/preview server
3. Server ready â†’ Poll for health/readiness
4. Browser verification â†’ Execute Playwright scenarios
5. Evidence capture â†’ Record results, screenshots
6. Completion â†’ Only after stages 4-5 succeed

### MUST
- Invoke `/playwright` skill after server is ready
- Run at least one browser scenario
- Capture `browser_verification` evidence in ExecutionResult

### MUST NOT
- Stop at "server started" without browser run
- Claim `success` without `browser_verification.executed: true`
- Skip browser verification without valid exception reason

### Exception Reasons (Valid)
- `playwright_unavailable`: Playwright tooling not available
- `environment_blocked`: Environment constraints prevent execution
- `browser_install_failed`: Browser installation failed
- `ci_container_limitation`: CI/container cannot run browser
- `missing_credentials`: Required credentials not available
- `deterministic_blocker`: Known blocker prevents meaningful verification
- `reclassified_noninteractive`: Feature reclassified as non-interactive

**If browser verification cannot run, MUST record exception_reason in ExecutionResult.**
```

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Over-enforcement for trivial changes | Clear classification model, explicit `frontend_noninteractive` path |
| Flaky browser tests | Structured evidence, blocked/failed distinction, readiness checks |
| Long-running server issues | Explicit state model, cleanup rules, non-blocking orchestration |
| Silent regressions in reporting | Reporting contract, acceptance tests for summary fields |
